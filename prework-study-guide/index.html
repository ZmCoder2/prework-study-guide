HTML

  • HTML stands for Hyper Text Markup Language.
  • HTML uses markup to describe and create the structure of web pages.
  • All HTML documents must start with a document type declaration. The HTML document type should look like the following: <!DOCTYPE html>.
    • The <!DOCTYPE> declaration represents the document type, and helps browsers to display web pages correctly.
    • Must only appear once, at the top of the web page (before any HTML tags).
    • Also, it is not case-sensitve because it is a declaration. Declarations are not case-sensitive; HTML elements are case-sensistve.
  • The structure of HTML is built up by elements and represented by tags.

HTML Elements:

  • HTML elements work together to build the structure of a webpage and provide a framework that makes it easy for a user to navigate the webpage.
  • Elements are a type of content on a webpage.
  • The following are types of elements that are commonly used to add content to web pages: Text elements, Image elements, Header elements, and List elements.
  • The <head> element contains information, or metadata, that helps the browser render the page correctly. Elements that are within the <head> are not visible when a user looks at a webpage.
    • Metadata is data about our webpage that the browser uses to effectively serve our webpage to users. Metadata is typically not seen by users, but it contains important information about accessibility, search engines, and performance.
  • The <body> element represents and contains the visible content shown to the user. All the elements that we want the user to and interact with must be place inside this element.
  • Inside the <head> element, we can use the following metadata tags:
    • The <title> element specifies a title for a webpage. This element is required and is extemely important for Search Engine Optimization (SEO).
      • Search engine optimization (SEO) is the process of improving the ranking of a website in search engines such as Google or Bing. Better SEO results in your website more frequently showing up in searches.
    • The <style> element defines the CSS styles associated with a website.
    • The <base> element defines the base URL for a webpage.
    • The <link> element connects an external resource to the HTML document.
    • The <meta> element defines metadata such as the character set, description, keywords, author, and viewpoint. Below, are some examples:
      • <meta charset="UTF-8" />. The <meta> element defines the charset, or character set, as UTF-8, which allows us to use most characters and symbols on the page.
      • <meta http-equiv="X-UA-Compatible" content="IE=edge">. The http-equiv element sets an HTTP header for content.
      • <meta name="vieport" content="width=device-width, initial-scale=1.0">. The viewport element sets the default view for our webpage.
    • The <script> element embeds client-side scripts in the HTML document.
    • The <p> element represents a paragraph or block of text.
      • You'll use this element extensively to wrap most of your text on your webpages.
  • HTML elements can be nested (this means that elements can contain other documents) and all HTML consist of nested HTML elements. They are called child elements.
    • If there are three elements nested inside of a <body> element, those three nested elements are known as child elements.
  • The element that holds all the children-in the previous case, the <body> element-is known as the parent element.
  • The "parent/child" language is often used by developers to explain the relationship among elements in HTML.
  • Elements are also used to divide the content on the page and to describe the content to broswers and other developers.
  • Below, are some common element types that are used to divide and describe a page:
    • Meta-related elements
    • Content division elements
    • Semantic elements.
      • In semantic HTML elements, the name of the elements describes its content and the role it plays on a webpage.
      • When possible, it is important to use semantic HTML elements, because they make our HTML easier to read by other developers and the browser.
      • Semantic HTML also plays an important role in making sure your webpage is accessible to the widest variety of users, including those who use assistive technologies such as screen readers.
    • Not all elements are semantic, though. It is common to see non-semantic elements such as <div> to create a division in the page or <span> to create an inline division.

HTML Tags:

  • Tags label pieces of content; Heading, Paragraph, and Table.
  • Tags usually come in pairs: a start tag and the end tag.
  • A start tag begins a section of a web page. It should look like the following:
    • <Tagname>
  • An end tag is writen just like the start tag but with a forward slash (/) inserted before the tag name. It should look like the following:
    • </Tagname>
  • Now, for how a start tag and end tag should look together; <Starttag>content goes here</Endtag>
  • The <html> tag represents the root, or start element of any HTML document.
    • You can put what is called an HTML attribute inside the <html> tag that defines additional information about an element and they are always defined in the beginning element, never the closing element.
    • For example; <html lang="en">. Inside the <html> tag is an attribute of lang="en", which tells the browser the language for the webpage (in the example, en for English).
  • The <h1> to <h6> tags are used to define HTML headings.
  • <h1> defines the most important heading. <h6> defines the least important heading.
  • Only use one <h1> per page and this should represent the main heading/subject for the whole page.
  • Also, do not skip heading levels - start with <h1>, then use <h2>, and so on.

CSS

  • CSS stands for Cascading Style Sheets.
  • CSS is used to add style to a webpage.
  • CSS is a programming language used by developers to define how webpage content is presented to users.
  • CSS defines color, aesthetics, visual layout, and orginization.
  • We use CSS to define style rules for specific elements of an HTML page and format them using a wide range of style options.
  • There are three ways to style a webpage using CSS:
    1. Inline CSS
    2. Internal CSS Style Sheet
    3. External CSS Style Sheet

Inline CSS:

  • Inline CSS is when a developer wants to style an element directly in the HTML file.
  • Inline styles apply to a specific HTML tag, using a style attribute with a CSS rule to style a particular page element only.
  • They are useful for quick and permenant changes but are less flexible than external and internal style sheets, because each inline style must be edited individually whenever you make a design change.
  • Example of Inline CSS is using the style attribute to change the color of a single h1 heading:
    • <h1 style="color:red;">Inline CSS Example</h1>

Internal Style Sheet:

  • Internal Style Sheets contain CSS rules for the webpage in the head section of the HTML file.
    • The rules only apply to that page, and you can configure CSS classes and ID's, which are used to identify elements, to style multiple elements in the page code.
    • A single change to the CSS rule will apply to all tagged page elements.
  • The following code shows what an Internal CSS Style Sheet should look along the lines like:
    • <head>
    • <style>
        • h1 {
        color:red;
        }
    • </style>
    • </head>
    • The preceding CSS states that all <h1> elements on the webpage will be red.
  • Inline CSS and Internal Style Sheets are totally acceptable methods under certain conditions, but the most common way to style a webpage is by using an External CSS Style Sheet.

External CSS Style Sheet:

  • External CSS Style Sheets are preferred in most cases because it allows developers to keep all their CSS rules in a seperate file, which makes design changes easier.
  • When using an External Style Sheet, we link the file using a <link> element placed in the HTML's <head>, as shown in the following code:
    • <head>
        • <link rel="stylesheet" href="./assets/style.css"/>
    • </head>

CSS Rule:

  • div {color: blue; text-align: justify;}
  • div is the selector.
  • color: and text-align: are the properties.
  • blue; and justify; are the values.
  • {color: blue; and text-align: justify;} are the declarations.
  • {color: blue; text-align: justify;} is the declaration block.
  • Declarations contain two important components: the CSS property we want to apply and the value of the property.
  • A margin indicates how much space we want around the outside of an element.
  • A padding indicates how much space we want around the content inside an element.
  • Example:
    • .card {
        • width: 80%;
        margin: 40px auto;
        border: 5px solid gray;
        box-shadow: 5px 10px #888888;
    • }
    • The period preceding card designates this as a class selector. Any HTML element assigned to the class, card, will gain these property assignments from the CSS.
    • The first declaration addresses the width of the element at 80% of the available width.
    • The margin property assigns 40px to the top and bottom, and the auto value to the left and right.
    • The border property assigns a border with a width of 5px that is solid and gray.
    • The box-shadow CSS property adds shadow effects around an elements frame.

Git

  • Git is a version control system that lets you manage and keep track of each version of your project.
  • Git is the most commonly used version control software among developers and a tool that you will likely use on the job.
    • Version control systems/softwares allows us to track and manage changes to our codebase. That way, if we need to make changes or even revert back to an earlier copy, we can do without losing other versions of our code.
    • Every time we save a version of our work using version control, it is assigned a unique identifier called an SHA, or hash, that allows us to keep a record of the specific changes committed.
      • This unique identifier provides a reference to that specific version of code so we can retrieve it as needed.
      • An SHA, or hash, is actually a long string of numbers or letters, like this: "e0eaed45609ae40ada29664384e679ada5d5004a". However, when using the SHA to access a version of their code, developers often type out only the first seven numbers or letters. This gives just the right amount of information to allow the right version to be retrieved and eliminates the need to copy the entire SHA.
  • To access Git, we use the command line in terminal.
    • Git commands always start with git and are used to save and track versions of your code.
    • Using Git commands, we can initiate or create a repo (repository), add our code to the repo, and view the different versions of our code, which are each stored in a unique commit.
  • git status: checks what branch we are currently on.
  • git checkout -b branch-name: creates a new branch and switches to it.

Git "Setup": Configuring user information used accross all local repositories

  • git config --global user.name "[firstname lastname]": sets a name that is identfiable for credit when review version history.
  • git config --global user.email "[valid-email]": sets an email address that will be associated with each history marker.
  • git config --global color.ui auto: sets automatic command line coloring for Git for easy reviewing.

Git "Setup and Init": Configuring user information, initializing and cloning repositories

  • git init: initializing an existing directory as a Git repository.
  • git clone [url]: retrieves an entire repository from hosted location via URL.

Git "Stage and Snapshot": Working with snapshots and the Git staging area

  • git add [file]: adds a file as it looks now to your next commit (stage).
  • git reset [file]: unstages a file while retaining the changes in working directory.
  • git diff: different of what is changed but not staged.
  • git diff --staged: different of what is staged but not yet committed.
  • git commit -m "[descriptive message]": commits your staged content as a new commit snapshot.

Git "Branch and Merge": Isolating work in branches, changing context, and integrating changes

  • git branch: lists your branches and a * will appear next to the currently active branch.
  • git branch [branch-name]: creates a new branch at the current commit.
  • git merge [branch]: merges the specified branch's history into the current one.
  • git log: shows all commits in the current branch's history.

Git "Inspect and Compare": Examining logs, diffs and object information

  • git log: shows the commit history for the currently active branch.
  • git log branchB..branchA: shows the commits on branchA that are not on branchB.
  • git show [SHA]: shows any object in Git in human-readable format.

Git "Tracking Path Changes": The versioning file removes and path changes

  • git rm [file]: deletes the file from project and stage the removal for commit.
  • git mv [existing-path] [new-path]: changes an existing file path and stage the move.
  • git log --stat -M: shows all commit logs with indication of any paths that moved.

Git Ignoring Patterns: Preventing unintentional staging or committing of files

  • logs/
    • *.notes : saves a file with desired patterns as gitignore with either direct string matches or wildcard globs.
    • pattern*/
  • git config --global core.excludesfile [file]: system wide ignores pattern for all local repositories.

Git "Share and Update": Retrieving updates from another repository and updating local repos

  • git remote add [alias] [url]: adds a git URL as an alias.
  • git fetch [alias]: fetches down all the branches from that Git remote.
  • git merge [alias]/[branch]: merges a remote branch into your current branch to bring it up to date.
  • git push [alias] [branch]: transmits local branch commits to the remote repository branch.
  • git pull: fetches and merges any commits from the tracking remote branch.

Git "Rewrite History": Rewriting branches, updating commits and clearing history

  • git rebase [branch]: applies any commits of current branch ahead of the specified one.
  • git reset --hard [commit]: clears staging area, and rewrites the working tree from specified event.

Git "Temporary Commits": Temporarily stores modified, tracked files in order to change branches

  • git stash: saves modified and staged changes.
  • git stash list: lists stack-order of stashed file changes.
  • git stash pop: writes working from the top of stash stack.
  • git stash drop: discards the changes from the top of stash stack.

GitHub

  • Git is great for tracking changes locally on your own computer, but in order for you to be able to collaborate with others, you need to use another tool, GitHub.
  • GitHub is a web-based interface that uses Git, the open source version control software that lets multiple people make seperate changes to web pages at the same time.
  • With GitHub, developers can build code, track changes, and innovate solutions to problems that might arise during the site development process simultaneously.

How to speak GitHub

  • Repository (repo)- a folder in which all files and their version histories are stored.
  • Branch- a workspace in which you can make changes that won't affect the live site.
  • Markdown (.md)- a way to write in Github that converts plain text to GitHub code. Sites such as Atom and Sublime Text are examples of free resources for developers using Markdown.
  • Commit Changes- a saved record of a change made to a file within the repo.
  • Pull Request (PR)- the way to ask for changes made to a branch to be merged into another branch that also allows for multiple users to see, discuss and review work being done.
  • Merge- after a pull request is approved, the commit will be pulled in (or merged) from one branch to another and then, deployed on the live site.
  • Issues- how work is tracked when using git. Issues allow users to report new tasks and content fixes, as well as allows users to track progress on a project board from the beginning to end of a specific project.
  • Federalists- a platform that securely deploys a website from a GitHub repository in minutes and lets users preview proposed and published changes.
